## Claude Code Harness Engineering

**27+ hook events** (updated May 2026). Handler TYPES: 5 (command, http, mcp_tool, prompt, agent).

Key events:
- SessionStart (additionalContext, initialUserMessage, watchPaths, reloadSkills) — injects context at every session start
- UserPromptSubmit — can block prompts before Claude processes them (prompt injection firewall)
- PostToolBatch — fires after batch of parallel tool calls
- PostToolUseFailure — fires after failed tool call (separate from PostToolUse)
- FileChanged — monitors file changes (e.g. .env modified externally)
- InstructionsLoaded — fires when CLAUDE.md / rules loaded

**`if` field on hook handlers** (confirmed):
- Permission rule syntax: `"if": "Edit(*.ts)"`, `"if": "Bash(rm *)"`, `"if": "Edit(.env*)"`
- Tool events only (PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied)
- Pre-filters hook execution — reduces attack surface of inline logic
- PreToolUse .env protection improvement: add `"if": "Edit|Write(.env*)"` to pre-filter before inline check runs

**Exit codes:**
- Exit 0: success; JSON parsed from stdout
- Exit 2: blocking error (for blockable events); stderr fed to Claude
- Other: non-blocking for most events

**PostCompact stdin**: Only `{session_id, transcript_path, cwd, hook_event_name}` — no `compacted_content`.

**`args[]` exec form** (v2.1.139, May 2026): Hook `"command"` accepts a string array (`["node", "-e", "..."]`) in addition to a shell string. Array form uses execve() directly — no shell interpretation, eliminates shell injection risk. Best practice for simple commands (`node -e`, `python3 -c`). Complex commands requiring pipes/conditionals still use string form or should be moved to wrapper scripts. **All 4 scaffold PreToolUse hooks updated to args[] form in v4.1.0.**

**SubagentStop event** (v2.1.139, May 2026): Fires when a subagent (spawned via `agent` tool or skill) completes. Distinct from `Stop` (which fires at end of main turn). Supports `decision: "block"` to intercept subagent output before Claude continues. Default `blocking: false`. Skills that spawn subagents should wire a `SubagentStop` hook to enforce output review.

**ConfigChange event** (v2.1.139, May 2026): Fires when settings.json or CLAUDE.md/AGENTS.md changes during a session. Can be used to re-validate harness integrity after config modifications (ASI02 defense).

**New events (v2.1.139, May 2026):** WorktreeCreate, WorktreeRemove (fire on worktree lifecycle — useful for cleanup hooks), MessageDisplay (fires when Claude outputs a message to the user — can be used for output auditing), Elicitation / ElicitationResult (fire when Claude requests user input and receives it — enables audit logging of all user decisions).

**skillListingBudgetFraction: 0.02** + **skillListingMaxDescChars: 1536** (default): Paired settings. `skillListingBudgetFraction` caps how much context budget skill listing can use. `skillListingMaxDescChars` caps per-skill description length (default 1536 chars). Set `skillListingBudgetFraction` in scaffold; omit `skillListingMaxDescChars` to use default unless a tighter cap is needed.

- RECOMMENDED: PreToolUse deny (exit 2) fires before permission rules and holds even in bypassPermissions mode; deny rules are deny-first — code.claude.com/docs/en/permissions

**Context management**
- RECOMMENDED: Project-root CLAUDE.md auto re-injected after compaction; invoked skills re-attach at 5,000 tokens each within a 25,000-token combined budget; keep CLAUDE.md <200 lines, procedures in skills + path-scoped `.claude/rules/*.md` — code.claude.com/docs/en/memory
- CONSENSUS: SessionStart with `compact` matcher is the established re-injection pattern; PostCompact exists for mid-session observability

**Security**
- RECOMMENDED: Layered .env defense: `permissions.deny` Read rules (gitignore semantics, any depth) + PreToolUse hook + sandbox. Deny rules cover built-in tools and recognized Bash file commands but NOT arbitrary subprocesses — only sandboxing blocks those — code.claude.com/docs/en/permissions
- CONSENSUS: Hook-based .env protection on Read|Edit|Write|Bash; instructions/.claudeignore are not security boundaries
- EMERGING: HTTP hooks for centralized team-wide hook policy (track only)

**Anti-patterns (community-converged):** flat `.md` in `.claude/skills/` (silently ignored); exit-2 Stop hooks without loop guards; treating CLAUDE.md/.claudeignore as security boundaries; argument-constraining Bash permission patterns (fragile per docs); CLAUDE.md >200 lines; trigger phrase buried at end of skill description; slow synchronous hooks on every PostToolUse/Stop.

### Rogue agent patterns / code smells
1. Missing Stop hook → no test enforcement; agent runs unconstrained
2. PostCompact without SessionStart backup → routing context lost on fresh sessions
3. PreToolUse without .env protection → secrets exposed (ASI02)
4. Over-broad `allowed-tools: Bash` → ASI02 violation
5. Missing "stop criteria" → agent loops (ASI08 Cascading Failures)
6. Ghost skill names in instructions → agent improvises on unknown skills (ASI01)
7. No harness.json → drift undetected
8. Skill body > 30 lines → instruction-dense, agent follows partial directive
9. Inline complex logic in hooks → bugs create security gaps; prefer `if` field + simpler logic
10. UserPromptSubmit not used → no prompt injection firewall at entry point
